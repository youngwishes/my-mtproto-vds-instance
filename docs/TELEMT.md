# Telemt

## Что это

[Telemt](https://github.com/nickvnlk/telemt) — высокопроизводительный MTProto-прокси сервер, написанный на Rust. Позволяет пользователям подключаться к Telegram через прокси-ссылки вида `tg://proxy?...`.

## Ключевые возможности

- **Fake-TLS маскировка** — режим TLS полностью имитирует реальное HTTPS-рукопожатие с указанным доменом (`tls_domain`). DPI видит легитимный HTTPS-трафик с достоверной цепочкой доверия, а сканеры получают настоящие ответы от маскировочного хоста.
- **Управление пользователями** — создание, удаление, ротация секретов, лимиты на IP/TCP/трафик/скорость через HTTP API.
- **Per-user ограничения** — уникальные IP, TCP-соединения, квоты трафика, rate-limiting, срок действия аккаунта.

## Роль в нашей системе

Telemt — конечный исполнитель. Этот FastAPI-сервис выступает прослойкой между Django-бэкендом и telemt API. Оба работают на одном VDS в Docker-контейнерах в общей сети.

```
FastAPI (порт 8080) ──HTTP──► telemt API (порт 9091) ──► telemt proxy (порт 443)
```

## API

HTTP API для управления прокси в runtime. Транспорт — HTTP/1.1, формат — JSON.

Базовый путь: `/v1`. Адрес задаётся через переменную `TELEMT_API_ROOT`.

### Формат ответа

Все ответы обёрнуты в конверт:

```json
{
  "ok": true,
  "data": { ... },
  "revision": "sha256-hex конфигурации"
}
```

### Используемые эндпоинты

Эндпоинты, к которым обращается этот сервис.

#### POST /v1/users

Создаёт пользователя на прокси.

```json
{
  "username": "string",
  "secret": "string (32 hex)",
  "max_unique_ips": 3
}
```

**Ответ:** `201` / `202` — данные пользователя и секрет.

**Ошибки:**

| Код | `error.code` | Условие | Поведение нашего сервиса |
|-----|-------------|---------|--------------------------|
| `409` | `user_exists` | Пользователь с таким `username` уже существует | Проксируется как `409 {"detail": "User already exists"}` |

> **Важно:** Django-бэкенд обязан гарантировать уникальность `username` перед вызовом этого эндпоинта.

---

#### GET /v1/users/{username}

Возвращает представление одного пользователя из конфига telemt. Используется `GetUserService` для проверки наличия пользователя и получения tg://-ссылки.

**Ответ:** `200` — пользователь существует (`UserInfo`).

> **Примечание:** сырой секрет в `UserInfo` **не возвращается** (только на `POST`/`rotate-secret`). `GetUserService` извлекает ссылку с секретом из поля `links` (приоритет `tls → secure → classic`).

**Ошибки:**

| Код | `error.code` | Условие | Поведение нашего сервиса |
|-----|-------------|---------|--------------------------|
| `404` | `not_found` | Пользователь с таким `username` не существует | Проксируется как `404 {"detail": "User not found"}` |

---

#### POST /v1/users/{username}/rotate-secret

Ротация секрета пользователя. Старый секрет перестаёт работать.

```json
{
  "secret": "string (32 hex)"
}
```

Тело необязательно — без `secret` telemt сгенерирует его автоматически.

**Ответ:** `200` / `202` — новый секрет.

> **Примечание:** этот эндпоинт в коде **не используется**. `RotateSecretService` вызывает `PATCH /v1/users/{username}` с полем `secret` — см. ниже.

---

#### PATCH /v1/users/{username}

Используется сервисом вместо `rotate-secret` — частичное обновление, передаётся только `{"secret": "..."}`.

```json
{
  "secret": "string (32 hex)"
}
```

**Ответ:** `200` / `202` — обновлённые данные пользователя.

**Ошибки:**

| Код | `error.code` | Условие | Поведение нашего сервиса |
|-----|-------------|---------|--------------------------|
| `404` | `not_found` | Пользователь с таким `username` не существует | Проксируется как `404 {"detail": "User not found"}` |

---

#### DELETE /v1/users/{username}

Удаляет пользователя и все связанные записи в access-картах.

Тело запроса отсутствует.

**Ответ:** `200` / `202`.

**Ошибки:**

| Код | `error.code` | Условие | Поведение нашего сервиса |
|-----|-------------|---------|--------------------------|
| `404` | `not_found` | Пользователь не существует | Ответ **игнорируется** — `RemoveUserService` не вызывает `raise_for_status()`. Массовое удаление идемпотентно: несуществующие пользователи молча пропускаются, клиент получает `204`. |

---

### Полезные эндпоинты для внедрения

Эндпоинты, которые не используются сейчас, но могут быть полезны в будущем.

#### GET /v1/users

Список всех пользователей с текущей статистикой (активные IP, соединения, трафик).

Применение: панель администратора, сверка пользователей между бэкендом и прокси.

---

#### PATCH /v1/users/{username}

Частичное обновление параметров пользователя (JSON Merge Patch):

| Поле                 | Тип            | Описание                       |
|----------------------|----------------|--------------------------------|
| `secret`             | string         | Новый секрет (32 hex)          |
| `max_tcp_conns`      | usize \| null  | Лимит TCP-соединений           |
| `max_unique_ips`     | usize \| null  | Лимит уникальных IP            |
| `expiration_rfc3339` | string \| null | Срок действия (RFC3339)        |
| `data_quota_bytes`   | u64 \| null    | Квота трафика в байтах         |
| `rate_limit_up_bps`  | u64 \| null    | Лимит скорости отдачи (байт/с) |
| `rate_limit_down_bps`| u64 \| null    | Лимит скорости загрузки (байт/с)|

Применение: гибкое управление тарифами (разные лимиты на разных планах), установка `expiration_rfc3339` напрямую вместо очистки по расписанию.

---

#### POST /v1/users/{username}/reset-quota

Сбрасывает счётчик потреблённого трафика пользователя.

Применение: обновление подписки (пользователь продлил — квота обнулилась).

---

#### GET /v1/stats/users/active-ips

Список пользователей с активными подключениями прямо сейчас.

Применение: мониторинг, выявление аномальной активности.

---

#### GET /v1/health/ready

Проверка готовности прокси: admission-gate открыт и есть хотя бы один здоровый upstream.

Применение: health-check в мониторинге, проверка перед выдачей ключа.

---

#### GET /v1/stats/summary

Сводная статистика: аптайм, количество соединений (всего / неудачных), ошибки по классам.

Применение: дашборд, алертинг при росте ошибок.
